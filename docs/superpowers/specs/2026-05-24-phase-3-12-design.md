# Phase 3.12 — Employee Permissions, Credentials & Notifications

**Date:** 2026-05-24  
**Status:** Approved

---

## Overview

Phase 3.12 adds three interconnected capabilities to the Employee module:

1. **Actionable permissions** — `reset_password` and `resign` permissions are enforced server-side, not just displayed
2. **Credential setup flow** — two paths based on whether the employee has an email address; IT is notified in-app and by email when action is required
3. **Permission inheritance from GroupRole** — a User's `role` field is kept in sync with the Role Template of their GroupRole, so permissions apply correctly on login

---

## 1. New Permissions

Three new permission keys are added to the `employees` module in `Permissions::catalog()`:

| Key | Purpose |
|-----|---------|
| `employees.reset_password` | Controls the Reset Password action on Employee Profile |
| `employees.resign` | Controls the Resign action on Employee Profile |
| `employees.set_credentials` | Allows setting Username/Password for employees without email; also marks who receives new-employee notifications |

Default grants:
- `admin` role: gets all three
- `hr` role: gets none (HR can add employees but cannot manage credentials or force-resign)
- `user` role: none

## 2. Employee Create → User Account Flow

Two paths depending on whether the employee record has an email address.

### Path A — Employee has email

1. `EmployeeService::create()` auto-creates a `User` record (already done in Phase 3.10):
   - `email` = employee email
   - `password` = employee code (e.g. `EMP-1042`)
   - `role` = role key from the employee's assigned GroupRole, or `'user'` if none
2. Dispatch `NewEmployeeNotification` to all users who have `employees.set_credentials` permission, with type `credentials_ready` and the default password in the payload.

### Path B — Employee has no email

1. No `User` is created at this point.
2. Dispatch `NewEmployeeNotification` with type `credentials_required` — IT must set username + password manually.
3. When IT sets credentials from the Employee Profile, `EmployeeService::createUserWithCredentials()` creates the `User` record:
   - `username` = value set by IT
   - `email` = null
   - `password` = value set by IT
   - `role` = role key from GroupRole, or `'user'`

## 3. Notification System

### Backend

- Use Laravel's built-in notification system (`php artisan notifications:table`).
- `User` model already uses `Notifiable` trait.
- Create `App\Notifications\NewEmployeeNotification` with:
  - **database channel** — persists to `notifications` table for in-app bell
  - **mail channel** — sends email (uses existing mail config from `mail_settings`; queued via Laravel Queue)
- Notification data payload:
  ```json
  {
    "type": "new_employee",
    "subtype": "credentials_ready | credentials_required",
    "employee_id": 42,
    "employee_name": "สมชาย ใจดี",
    "employee_code": "EMP-1042",
    "default_password": "EMP-1042"
  }
  ```
- Recipients: all `User` records where `hasPermission('employees.set_credentials')` is true.
- Dispatched inside `EmployeeService::create()` via `Notification::send($recipients, new NewEmployeeNotification($employee, $subtype))`.

### API

New endpoints under `auth:sanctum`:

| Method | URL | Description |
|--------|-----|-------------|
| `GET` | `/notifications` | List notifications for current user (paginated, unread first) |
| `PUT` | `/notifications/{id}/read` | Mark single notification as read |
| `PUT` | `/notifications/read-all` | Mark all as read |

### Frontend — Notification Bell

- Bell icon in the top Navbar with an unread-count badge (red dot, disappears at 0).
- Clicking opens a dropdown showing the latest 20 notifications.
- Each notification has a `View Employee →` button linking to the employee profile.
- "Mark all read" button at the top of the dropdown.
- Unread count polled every 60 seconds via React Query (`staleTime: 60_000`).

## 4. Employee Profile — Credentials Section

Visible only to users with `employees.set_credentials` permission.

**When employee has a linked User account:**
- Shows login method (email or username)
- "Reset Password" button (requires `employees.reset_password` permission) — opens a modal to set a new password
- Default password note if password has never been changed

**When employee has no linked User account (no email, no credentials set yet):**
- Form: Username + Password + Confirm Password
- "Create Account" button — calls new endpoint `POST /employees/{id}/credentials`
- On success, User is created; IT marks the notification as read manually (no auto-resolve)

### New API Endpoints

| Method | URL | Permission |
|--------|-----|-----------|
| `POST` | `/employees/{id}/credentials` | `employees.set_credentials` |
| `POST` | `/employees/{id}/reset-password` | `employees.reset_password` (already exists, now enforced) |
| `POST` | `/employees/{id}/resign` | `employees.resign` (already exists, now enforced) |

## 5. GroupRole → User.role Auto-Sync

**Rule:** a `User`'s `role` field always reflects the Role Template of the GroupRole the employee currently belongs to. If the employee is in no group, `User.role = 'user'`.

**Sync triggers:**

| Event | Action |
|-------|--------|
| Employee added to GroupRole | `User.role = group.role` |
| Employee removed from GroupRole | `User.role = 'user'` |
| GroupRole's role template changed | Update `User.role` for all member employees |
| New employee with default GroupRole | Assign to default group → `User.role = group.role` at creation |

Sync is performed inside `GroupRoleController` (store/update/destroy) and `EmployeeService::create()`.

---

## 6. Login — Username Support

`AuthController::login()` is extended to accept either `email` or `username`:

1. If the submitted field matches an email format → attempt `User::where('email', …)`
2. Otherwise → attempt `User::where('username', …)`
3. Validation requires one of `email` or `username`, plus `password`.

Frontend login form needs a minor label change: field placeholder/label updated to "Email or Username" to communicate that both are accepted.
